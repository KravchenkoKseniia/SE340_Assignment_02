# AFTER normalization

## The problem

```sql
SELECT customer_id, event_type, COUNT(*) AS events_count,
       MAX(event_time) AS last_event_time
FROM customer_events_wide
WHERE event_time >= NOW() - INTERVAL '180 days'
GROUP BY customer_id, event_type
ORDER BY events_count DESC
LIMIT 200;
```

`customer_events_wide` has 25 columns, but the analytics only touch 4: `event_id, customer_id, event_type, event_time`

## The fix

- `customer_events`
- `customer_events_details` linked 1:1 by `event_id` (FK)

Plus a covering index

```sql
CREATE INDEX idx_events_time_cust_type ON customer_events (event_time, customer_id, event_time);
```

## Before / After (Q4)

| |                       Before (wide) | After (core + covering index) |
|---|------------------------------------:|---:|
| Execution time |                            92.98 ms | 36.38 ms |
| Buffers |                               9,189 | 484 |
| Scan |          Seq Scan on 72 all 25 cols | Index Only Scan, Heap Fetches: 0 |

## Partitioning - tested and reverted

I also tried range-partitioning `customer_events` by `event_time` (one partition per year) to enable partition pruning.

- Q6 (90-day window) improved: it fits inside 2026, so the 2025 partition was pruned (`Subplans Removed: 1`).
- Q4 (180-day window) got worse (55 ms / 654 buf vs 36 ms / 484 buf): the
  window crosses into 2025, so both partitions stayed in the plan, and almost
  all of the 2026 partition matched -> the planner seq-scanned it instead of
  using the index.
